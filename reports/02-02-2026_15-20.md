**Objetivo**: Detalhar a técnica de "Native Core" sem dependências externas.

## 🛠️ A Técnica "Pure Core"

Em vez de importar bibliotecas pesadas (como `redis`, `kafkajs` ou `amqplib`), as implementações nativas do `@purecore` seguem a filosofia de comunicação direta via **Wire Protocol**.

### 1. Comunicação via TCP Puro
Utilizamos o módulo `net` do Node.js (totalmente suportado e otimizado no **Bun**) para abrir sockets TCP diretos. Isso elimina camadas de abstração desnecessárias e reduz o consumo de memória.

```typescript
// Exemplo de como fazemos o connect no RedisClient nativo
this.socket.connect(this.port, this.host, () => {
    this.isConnected = true;
    this.emit('connect');
});
```

### 2. O Parser de Protocolo (RESP para Redis)
Implementamos um parser manual para o **RESP (Redis Serialization Protocol)**. Ele é baseado em uma máquina de estados simples que processa chunks de strings (bufferizado) para lidar com a fragmentação de pacotes TCP.

- `+` : Simple Strings
- `-` : Errors
- `:` : Integers
- `$` : Bulk Strings
- `*` : Arrays

Isso garante que possamos ler e escrever no Redis com zero overhead de bibliotecas de terceiros.

### 3. Gerenciamento de Comandos (Request/Response Queue)
Como o TCP é um fluxo contínuo, implementamos uma `queue` de callbacks para casar a resposta do servidor com o comando enviado, garantindo que a lib seja assíncrona e segura.

### 4. Roadmap para NATS e Kafka
- **NATS**: Utiliza um protocolo baseado em texto simples (`PUB`, `SUB`, `MSG`). A implementação nativa seguirá o mesmo modelo de socket TCP + string parser.
- **Kafka**: Utiliza um protocolo binário complexo. A implementação nativa envolverá o mapeamento dos `API Keys` do Kafka para buffers binários customizados.

## 🧠 Por que essa abordagem?
- **Tamanho do Pacote**: O binário final (dist) é minúsculo.
- **Segurança**: Menos dependências = menor superfície de ataque (Supply Chain Attacks).
- **Performance**: Menos camadas de interpolação de tipos e abstrações.
